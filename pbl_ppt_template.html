Project Optic
Bare Metal, Bounded Intelligence

Core Mission: A dual-module C++ engine that proves hardware memory limits are not a barrier to geospatial analytics. The Base Engine processes standard imagery via a custom ThreadPool; the Out-of-Core Spike processes massive 16-bit aerospace datasets (e.g., Sentinel-2, 10980×10980 px, 242 MB) under a strict, mathematically verified 32 MB physical RAM constraint using POSIX mmap and the ST-BIT binary format.

The Problem: The Memory Wall

Standard heap allocation triggers an immediate Out-Of-Memory (OOM) failure when processing large GeoTIFFs (e.g., a 242 MB Sentinel-2 file) on constrained edge hardware with a 32 MB RAM target:

  • OOM Failure: Loading a large multispectral image into std::vector triggers immediate heap exhaustion on embedded edge hardware.
  • Cache Thrashing: Row-major disk layouts force the CPU to jump millions of bytes on the Y-axis, causing catastrophic I/O cache misses during out-of-core execution.
  • Heap Fragmentation: Dynamic new/malloc is fundamentally insufficient when the entire dataset exceeds physical RAM.

State-of-the-Art Analysis

Existing solutions load entire datasets into heap memory using standard allocators. This is viable for desktop-class RAM but causes immediate OOM failures on edge hardware constrained to 32 MB.

Our Innovation: ST-BIT + madvise

We flatten data into a Spatio-Temporal Binary Interleaved Tile (ST-BIT) format of 512×512 macro-blocks. POSIX mmap maps the file zero-copy; madvise(MADV_DONTNEED) evicts trailing pages per row, enforcing O(1) memory usage regardless of file size.

System Architecture

Layer 0

Storage Engine (Virtual Memory)

Uses POSIX mmap to create a zero-copy virtual memory map of the ST-BIT dataset. An eviction controller issues madvise(MADV_DONTNEED) signals to purge trailing pages per tile row, maintaining a sliding memory window strictly under 32 MB.

Layer 1

Compute Core (ThreadPool)

Image matrix is partitioned into horizontal bands (tasks) and distributed to a custom C++ ThreadPool. Worker threads apply spatial filters (Box Blur, Gaussian, Sobel) concurrently via std::mutex locks and std::condition_variable synchronization, guaranteeing zero race conditions.

Layer 2

Data Layout (ST-BIT)

The Base Engine stores pixels as a contiguous std::vector<Pixel> (1D flat array) with 2D coordinates flattened as Y * Width + X. The Spike uses 512×512 macro-blocks, aligning 2D spatial locality of the convolution kernel with the 1D physical locality of the SSD.

Layer 3

CLI Dispatcher (Dual-Mode)

A CLI routes execution via --mode [standard | out-of-core]. Standard mode activates the ThreadPool engine; out-of-core mode activates the Spike. Parameters include --input, --filter, and --threads N for pool sizing.

The Data Pipeline

1. Ingest & Map

CLI receives --input. Base Engine: stb_image decodes PNG/JPG into a contiguous heap array. Spike: Pre-flattened sentinel_raw.bin (ST-BIT, 242 MB) mapped zero-copy into virtual address space via mmap.

2. Stream & Compute

Base Engine: ThreadPool partitions the image into horizontal bands; workers apply spatial filters (Box Blur, Gaussian, Sobel, Sharpen) concurrently. Spike: Convolution kernel sweeps 22×22 tile grid; madvise(MADV_DONTNEED) evicts trailing tile rows per step.

3. Verify & Output

Processed matrix written to disk. Spike telemetry confirmed RAM stabilized at 31.5 MB (under 32 MB constraint). Cold read: 11.56 ms; hot (cached) read: 1.69 ms. Full 3×3 Box Blur over 10980×10980 px: 24.22 s.

Future Expansion: Architectural Directions

RISC-V & SIMD Acceleration

Compute Layer Optimization

As edge environments increasingly adopt open-source silicon, the engine will be optimized for the RISC-V ISA using the RVV (RISC-V Vector) extension, enabling massive SIMD convolutions directly at the hardware level to drastically reduce power consumption on remote UAV and satellite deployments.

AVX2 / NEON Hooks (Near-Term)

The contiguous std::vector<Pixel> layout and ST-BIT macro-blocks are structurally primed for SIMD. Maintainers can extend the Kernel class with AVX2 (x86) or NEON (ARM) intrinsics, processing multiple pixels per CPU cycle without altering threading logic.

RISC-V RVV (Long-Term)

Targeting the open-source silicon ecosystem for edge UAV deployments. The RVV extension will allow the engine to execute large-width SIMD convolutions natively, minimizing power draw on battery-constrained remote sensors.

io_uring & Computational Storage Drives

I/O Layer Evolution

While mmap page-faults control the RAM footprint, they block the CPU synchronously. Two directions eliminate this bottleneck entirely at different hardware layers.

  • Linux io_uring: Submit massive batches of async disk reads, prefetching ST-BIT tile rows into the CPU cache without blocking the convolution threads.
  • Modular I/O Backend: The codebase isolates the memory mapper, allowing a seamless swap from synchronous mmap to async APIs.
  • CSD Offloading: Push JIT decompression and convolution logic onto NVMe devices with embedded ARM/FPGA cores, bypassing the Memory Wall at the hardware boundary entirely.
