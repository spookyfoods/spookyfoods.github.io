Projct Optic
Bare Metal, Clothed Intelligence

Core Mission: To eliminate the I/O Wall in remote sensing by processing gigapixel-scale 16-bit multispectral data (Sentinel-2 L1C/L2A) faster than the disk can read it, utilizing a custom Zero-Copy architecture.

The Problem: Abstraction Overhead

Standard GIS tools (GDAL/Rasterio) rely on heavy abstractions and Python bindings. This creates significant bottlenecks when processing 100-megapixel images on edge hardware:

  • Memory Bloat: Loading a full 4GB TIFF into RAM causes immediate swapping on field laptops.
  • The GIL: Python's Global Interpreter Lock makes per-pixel manipulation prohibitively slow.
  • Cache Misses: "Vector of Vectors" (Jagged Arrays) destroys CPU cache locality.

State-of-the-Art Analysis

Existing solutions like GDAL utilize standard OS paging. Project Optic replaces this with manual virtual memory management to strictly control L1/L2 cache usage.

Our Innovation: Zero-Copy

We implement a "Rolling Ring Buffer" strategy. Instead of loading the image, we `mmap` the file and "decay" memory pages instantly after processing, ensuring O(1) memory usage regardless of image height.

System Architecture

Layer 0

Storage Engine (Virtual Memory)

Utilizes POSIX mmap with a custom Virtual Tile Manager. The "Rolling Ring Buffer" keeps only active scanlines in cache. As lines are processed, the memory is "decayed" (reused) for incoming lines.

Layer 1

Compute Core (SIMD)

Data is laid out in flat 1D arrays (std::vector<float>). We utilize manual AVX2 Intrinsics (_mm256_fmadd_ps) to process 8 pixels per CPU cycle. We replace standard 2D convolution with Separable Kernels (two 1D passes), reducing complexity by ~85%.

Layer 2

Memory Arena

Standard malloc/new causes heap fragmentation. We implement a Linear "Bump" Arena Allocator that pre-allocates a massive block at startup. Allocation is a simple integer increment; deallocation is instant.

Layer 3

Scheduler (Concurrency)

Implements a "Rolling Wave" Producer-Consumer model. Scout Threads prefetch raw bytes, while Builder Threads process tiles using Work-Stealing and an "N-2" Dependency Rule to handle halo rows without locking.

The Data Pipeline

1. Ingest & Map

CLI receives input.tif. Header parser reads IFD tags. mmap assigns file descriptor to virtual memory.

2. Stream & Compute (Pass 1 & 2)

Scout Thread loads rows into the Arena. Threads calculate row-wise sums (AVX), then column-wise sums using Rolling Buffer history.

Phase 7: Bare-Metal Edge ML

Planned Q4 2026

We will extend the C++ core to support inference-only execution of lightweight ML models directly on the edge device, bypassing heavy frameworks like TensorFlow.

Quantization Strategy

Weights will be compressed to 8-bit Integers (INT8). We will utilize AVX2 integer intrinsics (`_mm256_add_epi8`) to perform dot products, achieving ~4x throughput over FP32 while reducing memory bandwidth by 75%.

TinyML Architecture

Adoption of MobileNetV3-Small or custom auto-encoder backbones tailored for satellite tile classification. Models will be pruned to remove redundant connections before compilation.

OpticScript DSL (F#)

Planned Q1 2027

To replace complex JSON configs, we will implement a Type-Safe Domain Specific Language (DSL) using F#. This allows scientists to define processing pipelines with compiler-checked safety before execution.

  • Algebraic Data Types: Impossible to represent invalid pipeline states.
  • CLI Wrapper: Compiles DSL scripts into optimized arguments for the C++ engine.
  • Zero-Overhead: The DSL is strictly a configuration generator; runtime remains C++.

// Example OpticScript (F#)

pipeline {

input "sentinel_data.tif"

stage (Blur Kernel.Gaussian 5x5)

stage (Classify Model.TinyUrban)

output "result.geo"

}
